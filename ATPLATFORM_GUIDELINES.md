## Architecture Principles

- Each person should have their own Atsign, for example `@alice` or `@bob`.
- Agents and applications can either have a dedicated Atsign or share an Atsign with their owner, using namespaces for isolation.
- Multiple instances of the same agent can run under a single Atsign when coordinated.
- IoT devices can have their own Atsign for autonomous operation, or be managed as keys under a person's Atsign.
- All data is end-to-end encrypted by default. Only the Atsign owner holds the decryption keys.
- There is no application backend. atServers act as personal data stores and rendezvous points.
- Use namespaces to separate application data, for example `key.myapp@atsign`.
- Use `AtKey` with `sharedWith` to share encrypted data with specific Atsigns.
- Use `AtKey` with `metadata.isPublic = true` only for intentionally public data.
- Do not store application data in local files. Store data as AtKeys so it synchronizes across the owner's devices.

## Atsign Validation

- Always use `at_commons` to validate and parse every `String` used as an Atsign in Dart code.
- Example: `'@bob'.toAtsign()`.
- API parameters that expect Atsigns can accept plain strings, but validate them first.
- Use `.toAtsign()` exactly; the method name is case-sensitive.

## Initialization Pattern

- For Dart CLI and agents, initialize with `AtClientManager.getInstance().setCurrentAtSign(atSign, namespace, preferences)`.
- Set `AtClientPreference` with `rootDomain: AtRootDomain.atsignDomain`, `hiveStoragePath`, `commitLogPath`, and `namespace`.
- When running multiple agents or agent instances on the same machine, each must use a unique `hiveStoragePath` and `commitLogPath`, for example `Directory.systemTemp.createTempSync('agent_').path`.
- For agent/CLI authentication use `at_cli_commons`:

```dart
AtClient atClient = (await CLIBase.fromCommandLineArgs(args)).atClient;
```

- APKAM enrollment for new devices is supported via `EnrollmentService` in `at_client`.

## Flutter App Requirements

- Use `at_client_flutter`, not deprecated `at_onboarding_flutter`.
- Model auth/onboarding on the `at_client_flutter` example app:
  - https://github.com/atsign-foundation/at_client_sdk/tree/trunk/packages/at_client_flutter/example
- When using `file_picker`, add platform permissions:
  - macOS entitlements: `user-selected.read-only` and `user-selected.read-write` in Debug and Release.
  - iOS Info.plist usage descriptions.
  - Android `READ_EXTERNAL_STORAGE` and `WRITE_EXTERNAL_STORAGE` manifest permissions.

- For iOS, add `NSLocalNetworkUsageDescription` to Info.plist.

### Welcome / Auth Screen

The auth screen must support all four workflows:

1. Login from Keychain:
   `KeychainStorage().getAllAtsigns()` -> `AtSignSelectionDialog(existingAtSigns: list)` -> `AtAuthRequest(atSign, atKeysIo: KeychainAtKeysIo())` -> `PkamDialog` -> `setCurrentAtSign()`.
2. Onboard a New Atsign:
   `AtSignSelectionDialog.show(context)` -> `RegistrarCramDialog.show(context, authRequest as AtOnboardingRequest, registrar: registrar)` -> `CramDialog.show(context, request: authRequest, cramKey: cramKey)` -> `_setupAtClient()`. Use `RegistrarService(registrarUrl: 'my.atsign.com', apiKey: ...)` with the key supplied via `--dart-define=ATSIGN_REGISTRAR_API_KEY=...` (never commit the key).
3. APKAM enrollment:
   `AtSignSelectionDialog` -> `ApkamActivationDialog.show(atSign, rootDomain, appName, deviceName, namespaces)` -> wait for approval -> `AtAuthRequest(atSign, atAuthKeys: enrollmentResponse.atAuthKeys)` -> `PkamDialog` -> `setCurrentAtSign()`.
4. Login via `.atKeys` file:
   `AtKeysFileDialog.show()` -> `FileAtKeysIo` -> extract Atsign from filename -> `AtAuthRequest(atSign, atKeysIo: fileAtKeysIo)` -> `PkamDialog(backupKeys: [KeychainAtKeysIo()])` -> `setCurrentAtSign()`.

### Post-Auth Setup

- Get the app support directory.
- Create `AtClientPreference` with root domain, namespace, hive storage path, and commit log path.
- Call `AtClientManager.getInstance().setCurrentAtSign(atSign, namespace, prefs, atChops: response.atChops, atLookUp: response.atLookUp)`.

## Mandatory First-Run Atsign Gate

- Before showing any authentication UI, onboarding flow, login screen, or main application UI, implement a blocking First-Run Atsign Gate screen.
- This screen must appear on first launch if no Atsign is configured on the device.
- Check existing Atsigns via `KeychainStorage().getAllAtsigns()`.
- If empty, route to `AtsignGateScreen()`.
- The user must not access the app until they complete this step.
- Required title: `Using this app requires an Atsign.`
- Required Starter Pack URL: https://my.atsign.com/starterpack_app
- Primary button: `Get My Starter Pack`, opens the Starter Pack URL in the browser.
- Continue button: `Continue`, proceeds to the authentication screen.
- Persist a local flag only after at least one Atsign exists on the device.

## AtKey Construction

- Use namespaces for all application keys.
- Prefer structured construction:

```dart
final key = AtKey()
  ..key = 'devices.registry'
  ..namespace = 'myapp'
  ..sharedWith = '@other';
```

- `AtKey.shared('data', namespace: 'app')` may be used where suitable.
- Use `metadata.isPublic = true` only for intentionally public data.
- Use `Metadata()..immutable = true` for immutable mutex races.
- Add TTL to locks so abandoned work can be retried.

## Data Operations

### Async Communication

- Use `atClient.put()` to store data for another Atsign without requiring them to be online.
- To write directly to the cloud secondary, pass `putRequestOptions: PutRequestOptions()..useRemoteAtServer = true`.
- For reads, use `getRequestOptions: GetRequestOptions()..useRemoteAtServer = true` to read from the remote secondary or `..bypassCache = true` to skip cached values.

### Sync Communication

- Use `notificationService.notify()` with `NotificationParams.forUpdate(key, value: data)` to send data and wait for delivery confirmation.
- The sender receives a `NotificationResult`.
- Use when delivery confirmation matters.

### Notification / Event-Driven Communication

- Subscribe with:

```dart
atClient.notificationService
  .subscribe(regex: 'pattern', shouldDecrypt: true)
  .listen((notification) {});
```

- Send with `notificationService.notify(NotificationParams.forUpdate(key, value: data))`.
- Notifications are lightweight, support regex filtering, and reconnect on network loss.

### RPC

- Use `AtRpc` and `AtRpcClient` for structured request-response.
- Create an `AtRpc` with `AtRpcCallbacks` and an allowList.
- Requesters call `rpc.sendRequest(toAtSign: target, request: AtRpcReq(...))`.
- For multi-instance agents, use immutable mutex race or `ServiceFactoryWithNoOpSyncService()`.

### Streaming Data

- For continuous flows, use notification subscriptions and periodic or event-driven `notify()` calls.
- `atClient.startMonitor()` is deprecated; use `notificationService.subscribe()`.
- For streaming LLM answers, the pre-release `at_stream` package can be used.

## Multi-Agent Coordination

Pattern 1: Mutex race with immutable keys.

- First instance writes an immutable key with `Metadata()..immutable = true`.
- Use `putRequestOptions: PutRequestOptions()..useRemoteAtServer = true`.
- The instance whose `put()` returns true handles the task.
- Other instances skip.
- Use TTL to auto-release stale locks.

Pattern 2: `ServiceFactoryWithNoOpSyncService`.

- Create agent instances with ephemeral per-session storage and no sync.
- Use for stateless request-response agents that do not need persisted state across restarts.

## Networking File Transfer & Access Control

### TCP Tunnels

- Use NoPorts via `noports_core` for end-to-end encrypted TCP connections with zero open listening ports.
- The `npt` CLI and `Npt` class create local tunnel ports routed through the Atsign Platform.
- References: https://pub.dev/packages/noports_core and https://docs.noports.com/

### Encrypted File Transfer

- For bulk transfer, encrypt the file client-side.
- Upload the encrypted file to a web file service.
- Send the URL and encryption key as an encrypted AtKey to the recipient.
- The recipient downloads and decrypts locally.

### Policy Enforcement

- Use NPA/NPP policy patterns for access control where network access must be authorized.
- Keep policy decisions as explicit AtKeys or policy service records.
- Share policy state only with the Atsigns that need to enforce or audit it.

## Key Conventions

- Use lowercase, dot-delimited key names.
- Prefix by domain area, for example `device.<id>.record`, `telemetry.<id>.<timestamp>`, `alert.<id>`.
- Keep namespace stable for the app.
- Include timestamps or UUIDs for append-only event keys.
- Store lists only when the list is the actual synchronized state; otherwise prefer per-record keys plus an index key.
- Validate every Atsign before assigning `sharedWith`.

## Error Handling

- Wrap AtClient operations in try-catch.
- Common exceptions include `AtClientException`, `AtKeyException`, and `AtConnectException`.
- Check `AtClientManager.getInstance().atClient.syncService.syncInProgress` before critical operations.
